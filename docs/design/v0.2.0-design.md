# TechLead v0.2.0 设计方案

## 目标

让 Codex / Claude 能够执行**复杂、长时间、可能超出上下文限制**的编程任务，通过外部驱动的循环机制确保任务持续推进。

---

## 核心设计原则

1. **外部驱动循环**：CLI 控制执行节奏，不是 agent 自己决定继续或停止
2. **文件系统范式**：所有数据存储为文本文件（Markdown/YAML/JSON），人类可读、Git 友好
3. **状态即字段**：任务状态用 JSON 字段表示，不是文件夹位置
4. **三级上下文**：L0/L1/L2 模型优化 Token 使用

---

## task.json 结构

```json
{
  "id": "T-002",
  "title": "实现 OAuth 登录",
  "status": "in_progress",
  "phase": "plan",
  "created_at": "2025-02-28",
  "started_at": "2025-02-28",
  "completed_at": null
}
```

### 状态字段

- **status**: `backlog` | `in_progress` | `review` | `testing` | `done` | `failed`
- **phase**: `plan` | `exec` | `review` | `test` | `completed`

`phase` 决定 task 文件夹中存在哪些子文件夹。

## 三级上下文模型（L0/L1/L2）

借鉴 OpenViking 的设计，优化 Token 使用：

| 层级 | 文件 | Token 限制 | 用途 |
|------|------|-----------|------|
| **L0** | `.abstract.md` | ~100 tokens | 快速筛选、向量搜索 |
| **L1** | `.overview.md` | ~2k tokens | 内容导航、理解范围 |
| **L2** | 原始文件 | 无限制 | 完整内容、按需加载 |

### 应用示例

```
tasks/T-002-实现OAuth登录/plan/
├── .abstract.md          # L0: "实现 GitHub OAuth 登录，支持 state 防 CSRF"
├── .overview.md          # L1: 计划章节导航
├── discussion.md         # L2: 完整多角色讨论
└── plan.md               # L2: 详细执行计划
```

---

## 状态机

```
Backlog -> Plan -> In Progress -> Review -> Testing -> Done
                              ^         |            |
                              +---------+------------+
                                        |
                                     Failed
```

### 阶段说明

1. **Backlog**: 待办，只有 README.md
2. **Plan**: 多角色讨论，生成 plan/discussion.md 和 plan/plan.md
3. **In Progress**: 执行阶段，有 work-log.md 记录每步操作
4. **Review**: 对抗性审查，review/ 目录存放 reviewer 输出
5. **Testing**: 对抗性测试，test/ 目录存放测试结果
6. **Done**: 完成归档
7. **Failed**: 失败后回到 In Progress 重新执行

## CLI 设计

```bash
# 初始化项目
techlead init

# 添加任务（创建 backlog 任务）
techlead add "实现 OAuth 登录"
# → 创建 tasks/T-004-实现OAuth登录/
# → 生成 task.json (status: backlog)
# → 生成 README.md

# Plan 阶段
techlead plan T-004 --roles architect,security,dx
# → 状态改为 in_progress, phase: plan
# → 创建 plan/discussion.md（多角色讨论）
# → 创建 plan/plan.md（最终计划）

# 开始执行
techlead start T-004
# → phase 改为 exec
# → 创建 work-log.md
# → 更新 current.json

# 执行一轮（外部驱动）
techlead step
# → 读取 current.json
# → 调用 codex/claude 执行一步
# → 更新 work-log.md

# 提交 Review
techlead review T-004
# → 状态改为 review
# → phase 改为 review
# → 创建 review/reviewer-a.md（对抗性审查）

# 提交测试
techlead test T-004
# → 状态改为 testing
# → phase 改为 test
# → 创建 test/adversarial-test.md

# 标记完成
techlead done T-004
# → 状态改为 done
# → phase 改为 completed

# 或使用编排命令（自动组合上述分阶段命令）
techlead run
techlead loop --max-cycles 20 --max-no-progress 3

# 查看当前状态
techlead status
# → 读取 current.json
# → 显示当前任务、最近日志

# 列任务
techlead list
# → 列出所有 tasks/ 下的任务
```

## 核心特性

### 1. Plan 阶段（多角色规划）

任务开始前，进行多角色讨论：
- **Architect**: 系统边界、架构决策
- **Security**: 安全风险、密钥管理
- **DX**: 开发者体验、API 设计

每个角色独立分析，然后讨论达成共识，输出 plan.md 作为执行依据。

### 2. 对抗性 Review

任务完成后，引入独立 reviewer：
- 不看原始 plan，只看 diff
- 从特定视角审查（security、performance、future-user）
- 输出结构化报告：CRITICAL / WARNING / PASS
- CRITICAL 必须修复才能进入下一阶段

### 3. 对抗性测试

除了常规测试，引入对抗性测试：
- 测试者扮演攻击者或挑剔用户
- 覆盖边界情况、误用场景、安全攻击向量

### 4. 双 Agent 支持

同时支持 Claude Code 和 Codex：
- 通过 CLI 统一适配，屏蔽底层差异
- 不依赖特定 agent 的专有功能

## 与原版 TechLead 的差异

| 原版 | v0.2.0 |
|------|--------|
| 6 状态 sprint 机 | 简化状态机 |
| HTTP 验收测试 | 本地 shell 测试 |
| 400 行 protocol | 简洁 CLI 驱动 |
| Multi-perspective review | 对抗性 review |
| 复杂并行 runner | 先串行，并行后期优化 |
| 向量数据库 | 文件系统范式 |

## 使用方法

```bash
# 编译
pnpm run build

# 使用 CLI（分阶段）
node ./dist/cli.js init
node ./dist/cli.js add "Your task"
node ./dist/cli.js plan T-001
node ./dist/cli.js start T-001
node ./dist/cli.js step T-001
node ./dist/cli.js review T-001
node ./dist/cli.js test T-001
node ./dist/cli.js done T-001

# 或使用编排模式
node ./dist/cli.js loop --max-cycles 20 --max-no-progress 3
```
